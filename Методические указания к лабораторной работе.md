# 13 мая 2026 года. Группы ПИ302 и 3ПИ401

### МЕТОДИЧЕСКИЕ УКАЗАНИЯ К ПРАКТИЧЕСКОЙ РАБОТЕ

**Тема:** Проектирование аналитических панелей (Dashboards) в среде «Навигатор BI» (СБЕР) на базе синтетических данных.
**Компетенции ФГОС:** 
*   **ПК-1:** Способность собирать, обрабатывать и анализировать большие массивы данных с использованием современного инструментария.
*   **ПК-3:** Способность проектировать архитектуру хранилищ данных и BI-систем с учетом требований информационной безопасности.

**Цель работы:** 
1. Сгенерировать валидные датасеты с контролируемыми аномалиями (до 1 000 000 записей) с помощью утилиты `datagen-go`.
2. Спроектировать витрины данных (Data Marts).
3. Разработать дашборды в «Навигатор BI» для выявления паттернов и аномалий.
4. Имплементировать политики безопасности (Row-Level Security, Data Masking).

**Инфраструктурный стек:**
*   **Генерация данных:** `datagen-go` (CLI).
*   **СУБД (DWH Layer):** PostgreSQL 15+ (для промежуточного хранения и применения политик безопасности перед подачей в BI).
*   **BI-система:** Навигатор BI (СБЕР) — учебная версия.

### СЦЕНАРИЙ 1: E-commerce — Анализ эффективности маркетинга и сдвигов тренда

**Бизнес-контекст:** Анализ продаж розничной сети. Необходимо визуализировать влияние маркетинговой кампании (сдвиг тренда) и сезонные колебания спроса.
**Ограничение:** 500 000 записей.

#### 1. Генерация данных
Создайте базовый файл `data/samples/base/sales_sample.csv` со структурой: `transaction_id, date, region, product_category, sales_amount, discount`.

Конфигурация `config.yaml` для `datagen-go`:
```yaml
output_file: "data/generated/retail_sales.csv"
rows: 500000
sample_file: "data/samples/base/sales_sample.csv"
anomalies:
  # Имитация эффекта от маркетинговой кампании (рост продаж)
  - type: "trend_shift"
    column: "sales_amount"
    start_row: 150000
    end_row: 350000
    magnitude: 25.0
  # Имитация сезонности (например, колебания по дням недели)
  - type: "seasonal_pattern"
    column: "sales_amount"
    period: 7000
    magnitude: 0.3
```
*Выполнение:* `.\datagen.exe -config config.yaml`

#### 2. Проектирование дашборда в Навигатор BI
*   **Источник данных:** Подключение к PostgreSQL (таблица `retail_sales`) или прямой импорт CSV.
*   **Визуализация 1 (Line Chart):** Динамика `sales_amount` по оси времени (`date`). Цель: визуальная фиксация `trend_shift` на интервале строк 150k-350k.
*   **Визуализация 2 (Bar Chart):** Суммарные продажи в разрезе `region`.
*   **Визуализация 3 (KPI / Indicator):** Средний чек (Average `sales_amount`) и сравнение с предыдущим периодом.

#### 3. Имплементация защиты (RLS)
Требование: Региональные менеджеры должны видеть данные только своего региона.
Реализация на уровне PostgreSQL (до передачи в BI) или через внутренние фильтры безопасности Навигатор BI:
```sql
-- Создание политики RLS в PostgreSQL
ALTER TABLE retail_sales ENABLE ROW LEVEL SECURITY;

CREATE POLICY region_isolation_policy ON retail_sales
    USING (region = current_setting('app.current_region'));

-- В Навигатор BI настраивается маппинг ролей пользователей на переменную app.current_region
```

### СЦЕНАРИЙ 2: FinTech — Мониторинг транзакций и Anti-Fraud

**Бизнес-контекст:** Выявление подозрительных финансовых операций. Поиск единичных выбросов (фрод) и аномальных корреляций между суммой транзакции и комиссией.
**Ограничение:** 800 000 записей.

#### 1. Генерация данных
Создайте базовый файл `data/samples/base/tx_sample.csv` со структурой: `tx_id, timestamp, user_id, merchant_type, amount, fee, status`.

Конфигурация `config.yaml` для `datagen-go`:
```yaml
output_file: "data/generated/fin_transactions.csv"
rows: 800000
sample_file: "data/samples/base/tx_sample.csv"
anomalies:
  # Имитация крупной мошеннической транзакции
  - type: "statistical_spike"
    column: "amount"
    position_row: 450000
    magnitude: 500.0
  # Имитация сбоя в расчете комиссии (аномальная корреляция)
  - type: "multi_column_correlation"
    column: "amount"
    target_column: "fee"
    magnitude: 0.15
```
*Выполнение:* `./datagen_linux -config config.yaml`

#### 2. Проектирование дашборда в Навигатор BI
*   **Визуализация 1 (Scatter Plot / Диаграмма рассеяния):** Ось X — `amount`, Ось Y — `fee`. Цель: выявление кластера нормальной корреляции и отклонений, сгенерированных через `multi_column_correlation`.
*   **Визуализация 2 (Table / Flat Table):** Детализация транзакций с условным форматированием (Conditional Formatting). Строки, где `amount` превышает 99-й перцентиль, подсвечиваются красным (выявление `statistical_spike`).
*   **Визуализация 3 (Pie Chart):** Распределение транзакций по `merchant_type`.

#### 3. Имплементация защиты (Data Masking)
Требование: Идентификаторы пользователей (`user_id`) относятся к PII (Personal Identifiable Information) и должны быть замаскированы для аналитиков первой линии.
```sql
-- Создание представления (View) с маскированием данных для подключения в Навигатор BI
CREATE VIEW vw_fin_transactions_secure AS
SELECT 
    tx_id,
    timestamp,
    CONCAT(SUBSTRING(user_id, 1, 4), '****', SUBSTRING(user_id, LENGTH(user_id)-3, 4)) AS user_id_masked,
    merchant_type,
    amount,
    fee,
    status
FROM fin_transactions;
```

### КРИТЕРИИ ПРИЕМКИ (Definition of Done)

1. **Инженерия данных:** Сгенерировано два CSV-файла объемом 500k и 800k строк. Файлы успешно загружены в СУБД/BI без потери точности типов данных (Data Type Integrity).
2. **Аналитика:** Дашборды в Навигатор BI содержат минимум 3 визуализации каждый. Аномалии (`trend_shift`, `statistical_spike`, `multi_column_correlation`) явно идентифицируются на графиках.
3. **Безопасность:** 
   * Продемонстрирована работа RLS (пользователь с ролью "Менеджер региона А" не видит данные "Региона Б").
   * PII данные (`user_id`) замаскированы на уровне источника данных или средствами BI-системы.
4. **Производительность:** Время рендеринга дашборда при фильтрации по всему массиву (до 1 млн записей) не превышает 3 секунд (требует настройки индексов в СУБД или использования in-memory движка BI).
